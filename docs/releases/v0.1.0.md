# basis-gateway v0.1.0 Release Notes

**Release date:** 2026-06-08
**Status:** Initial public release

---

## Overview

`basis-gateway` v0.1.0 is the first public release of the HTTP enforcement boundary for the BASIS ecosystem. It authenticates callers via OIDC/JWT, normalizes identity from verified token claims, delegates authorization decisions to `basis-core`, enforces the returned decision at the HTTP boundary, and emits structured audit evidence for every outcome.

This release establishes the core authorization path and the audit model. It is a pre-production proof of concept intended for integration testing, evaluation, and development use.

---

## What's in this release

### Authentication and identity

- Bearer token verification supporting RS256, RS384, RS512, ES256, ES384, ES512; `alg=none` rejected unconditionally
- OIDC discovery (`{issuer}/.well-known/openid-configuration`) with optional JWKS URI override (`OIDC_JWKS_URI`)
- In-memory JWKS cache with configurable TTL (`JWKS_CACHE_TTL_SECONDS`, default: 300s)
- Subject normalization: `sub` → `subject_id`, `preferred_username` → `name`, `realm_access.roles` or `roles` → roles, standard JWT attributes

Subject identity is always derived from the verified token. The `EvaluateRequest` schema rejects any request body containing `subject_id` or `subject_roles`.

### Authorization evaluation

- `POST /v1/evaluate` — authenticated requests delegated to `basis-core` `EnforcementPoint`; gateway enforces the returned decision (ALLOW → 200, DENY → 403)
- JSON policy file loaded once at startup (`POLICY_PATH`); service will not become ready if missing or invalid
- Optional policy version provenance in responses and audit records (`POLICY_VERSION`)
- Fail-closed on all error paths: unexpected evaluation errors produce DENY, not ALLOW

### Audit evidence

- Kernel decision events (`AuditEvent`, `schema_version: "1.0"`) written by `basis-core` for every evaluation
- Gateway-level events for all significant pre-evaluation outcomes:
  - `gateway.evaluation_requested` — authenticated request, about to call kernel
  - `gateway.authentication_failed` — token missing, malformed, expired, issuer/audience mismatch, JWKS failure
  - `gateway.validation_failed` — request body fails schema validation
  - `gateway.evaluator_unavailable` — evaluator not initialized at request time
  - `gateway.evaluation_failed_closed` — unexpected exception during evaluation
- All audit records carry the same `correlation_id` as the `X-Correlation-ID` response header

### Correlation IDs

- UUIDv4 generated per request by `CorrelationMiddleware` before any route handler runs
- `X-Correlation-ID` header present on all responses (200, 400, 401, 403, 500, 503)
- Same correlation ID in response body, response header, and all audit records for that request
- Caller-supplied `X-Correlation-ID` headers are ignored; the gateway is the authoritative source

### Operational readiness

- Per-component readiness probe (`GET /ready`): `configuration_loaded`, `oidc_configured`, `jwks_available`, `policy_loaded`, `audit_writer`, `evaluator_initialized`
- Liveness probe (`GET /health`) — always responds while the process is running
- Configurable audit failure escalation: readiness degradation (Model B, default) or strict fail-closed (`AUDIT_FAIL_CLOSED=true`, opt-in)
- Automatic audit recovery: first successful write after degradation restores readiness without restart

## What is not in this release

The following are intentionally out of scope. See [`docs/release-readiness.md`](../release-readiness.md) for the full list and rationale.

- Persistent audit storage (current backend: `LogAuditWriter` to process log)
- Metrics, Prometheus, OpenTelemetry, or distributed tracing
- Docker, docker-compose, or Kubernetes manifests
- Dynamic policy reload
- `basis-console` or `basis-adapters` integration
- Horizontal scaling or multi-instance coordination

---

## Known limitations

- **Static policy load**: policy loaded once at startup; changes require a process restart
- **Log-backed audit only**: no durable storage, no guaranteed delivery, no tamper-evidence
- **In-process JWKS cache**: not shared across instances; key rotation depends on TTL
- **Single-instance only**: audit thresholds and readiness state are per-process; multi-instance deployments are untested
- **Role claim conventions**: supports Keycloak-style `realm_access.roles` and flat `roles`; other IdP structures may require changes to `subject_mapper.py`
- **`RequestValidationError` handler gap**: a latent audit event gap exists in `main.py` if the `/v1/evaluate` route signature is changed to use FastAPI-managed body parameters (currently unreachable). See [`docs/audit-model.md`](../audit-model.md) §9.

## Architecture invariants confirmed

- **Kernel evaluates. Gateway enforces. Audit records evidence.** `basis-core` `EnforcementPoint` owns evaluation semantics; the gateway does not reinterpret decisions.
- **Subject identity from token only.** Identity is never accepted from the request body.
- **Fail closed on every error path.** Errors deny; audit failures do not alter authorization decisions.
- **Correlation IDs are gateway-generated.** External parties cannot influence the audit trail via request headers.
